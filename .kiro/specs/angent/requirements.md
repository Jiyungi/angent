# Requirements Document

## Introduction

Angent is a self-improving, goal-driven deal-sourcing agent built for solo angel investors who are priced out of enterprise tools like PitchBook and Harmonic. The investor states an investment thesis and a measurable goal; Angent then runs a continuous control loop that discovers newly-emerging startups from public signals, qualifies them against the thesis, drafts personalized outreach emails, sends them only after human approval, and learns from reply outcomes to improve its qualification and targeting over time.

The product's differentiator is not higher cold-email volume (a saturated, declining channel) but a self-improving outreach loop in which the qualification/scoring model improves from real reply data while email volume is kept low and quality high.

The system is architected as a goal-driven control loop (not a one-pass pipeline) coordinated by a Planner that re-decides each iteration based on progress. Six cooperating agents (Planner, Scanner, Qualifier, Writer, Sender, Optimizer) operate over a shared ClickHouse blackboard. The loop terminates when the goal is met, the time budget elapses, or the email budget is spent.

This document defines the requirements for a hackathon-scope build (target ~7 hours, one person) that must run end-to-end on its own Python core, with sponsor integrations (Guild governance, Pioneer scoring, Airbyte Gmail sender, OpenUI generative UI, Langfuse tracing) layered on with explicit fallbacks so that no single integration is a blocker.

## Glossary

- **Angent**: The overall self-improving deal-sourcing agent system described in this document.
- **Investor**: The solo angel investor who is the human operator and approver.
- **Thesis**: The investor-provided description of the kinds of startups they want to invest in, used as the qualification target.
- **Goal**: A structured objective `{ target_metric, deadline, email_budget }` that the loop pursues. `target_metric` is a measurable outcome target (for example a reply rate or count of contacted thesis-fit companies), `deadline` is a wall-clock time budget, and `email_budget` is a hard cap on emails that may be sent.
- **Control_Loop**: The repeating goal-driven iteration owned by the Planner; each repetition is a Tick.
- **Tick**: A single iteration of the Control_Loop.
- **Planner**: The agent that owns the Goal and decides each Tick what action to take (widen thesis, change email angle, send more, stop).
- **Scanner**: The agent that pulls candidate startups from public signal sources.
- **Qualifier**: The agent that scores and reasons about whether a candidate fits the Thesis.
- **Writer**: The agent that drafts personalized outreach emails for qualified candidates.
- **Sender**: The agent that sends approved emails through a sender backend.
- **Optimizer**: The agent that collects reply and open outcomes and feeds learning.
- **Pioneer**: The Fastino adaptive-inference scoring model used only for the self-improving qualification classifier; optional and pluggable.
- **Heuristic_Scorer**: The built-in fallback scorer used when Pioneer is unavailable.
- **TrueFoundry_Gateway**: The OpenAI-compatible LLM gateway used for all reasoning and email prose.
- **ClickHouse**: The database used as the shared state blackboard and outcome analytics store, holding the `companies` and `emails` tables.
- **Signal_Source**: A public data source of emerging-startup signals. Confirmed sources are GitHub (via Airbyte) and Hacker News (direct API); Hugging Face Hub is a stretch source.
- **Airbyte_Agent**: The Airbyte Agents connector layer used for GitHub discovery and, if a tier unlock occurs, Gmail sending.
- **SMTP_Sender**: The default email-sending backend implemented at `angent/email_sender.py` using a Gmail App Password.
- **Gmail_Agent_Sender**: The alternate email-sending backend via the Airbyte Gmail connector, available only if the Airbyte tier unlocks.
- **Sender_Interface**: The common interface that both SMTP_Sender and Gmail_Agent_Sender implement.
- **Guild_Orchestrator**: The TypeScript Guild agent that orchestrates the loop and enforces governance (human-approval gate, rate limit, email-budget cap).
- **Governance_Gate**: The human-approval and policy-enforcement checkpoint that all sends must pass.
- **OpenUI_Surface**: The runtime generative UI produced via OpenUI Lang for agent-decided surfaces (the thesis-refinement chat and the on-demand deep-dive view).
- **React_Shell**: The plain, statically authored React components (dashboard cards, drafted-email preview, loop-status) that are not generated by OpenUI.
- **Impeccable**: The external polish tool (`npx impeccable detect`) permitted to touch only the React_Shell.
- **Langfuse**: The observability tool used to trace every agent step and LLM call.
- **Builder**: The single developer who designs, implements, tests-by-running, commits, and demos the entire Angent system end to end.
- **Repository**: The GitHub repository at https://github.com/Jiyungi/angent that holds all Angent code; the `main` branch is the single source of truth.
- **Publisher**: The component that serializes a run's qualified companies into a Deal_Memo and publishes it to cited.md via Senso.
- **Senso**: The sponsor service used to publish the Deal_Memo to cited.md; authenticated via SENSO_API_KEY sent as the X-API-Key header against base URL https://apiv2.senso.ai/api/v1.
- **cited.md**: The open-web destination where the published Deal_Memo is served at cited.md/<handle>/<slug> as human HTML plus an agent-native payload.
- **Deal_Memo**: The published markdown artifact summarizing the run's qualified companies, each with a provenance citation.
- **Payment_Gate**: The x402 paywall protecting the deal-memo fetch endpoint, charging agents per fetch.
- **x402**: The agent payment-rails protocol used to require and settle per-fetch payments in USDC on the Base Sepolia testnet.
- **Facilitator**: The x402 facilitator service (https://x402.org/facilitator) that settles payments via EIP-3009 so the payer is gasless.

## Requirements

### Requirement 1: Goal definition and loop initiation

**User Story:** As an Investor, I want to set a thesis and a measurable goal with a deadline and an email budget, so that Angent pursues my objective autonomously within limits I control.

#### Acceptance Criteria

1. WHEN the Investor submits a Thesis of 1 to 5,000 characters together with a Goal containing target_metric, deadline, and email_budget, THE Angent SHALL accept the Goal.
2. WHEN a Goal is accepted, THE Angent SHALL persist the Goal and start a Control_Loop.
3. IF a submitted Goal is missing target_metric, deadline, or email_budget, THEN THE Angent SHALL reject the Goal without persisting it and return a message identifying the missing field.
4. IF a submitted email_budget is outside the range 1 to 1,000, or a submitted deadline is not between 1 minute and 365 days in the future, THEN THE Angent SHALL reject the Goal without persisting it and return a validation message identifying the invalid field.
5. WHEN a Goal is accepted, THE Angent SHALL record the Goal, the start time, and the initial loop state in ClickHouse within 2 seconds and before the first Tick executes.
6. IF recording the accepted Goal and initial loop state in ClickHouse fails, THEN THE Angent SHALL not start the Control_Loop, SHALL not leave a partial record, and SHALL return an error indication that initialization did not complete.

### Requirement 2: Goal-driven control loop and per-Tick planning

**User Story:** As an Investor, I want Angent to re-decide its strategy every iteration based on progress, so that it adapts instead of running a fixed one-pass pipeline.

#### Acceptance Criteria

1. WHILE the Goal is not met AND the elapsed time is within the configured time budget AND the number of emails sent is below the configured email_budget, THE Control_Loop SHALL execute Ticks repeatedly, running at most one Tick at a time.
2. IF the Goal is met, the time budget is elapsed, or the email_budget is spent, THEN THE Control_Loop SHALL stop executing Ticks and record the termination reason.
3. WHEN a Tick begins, THE Planner SHALL read the current loop state and the Goal and produce a plan for that Tick.
4. WHEN the Planner produces a plan, THE Control_Loop SHALL execute Scanner, then Qualifier, then Writer, then Sender, then Optimizer in that order using the plan.
5. IF any pipeline stage (Scanner, Qualifier, Writer, Sender, or Optimizer) fails during a Tick, THEN THE Control_Loop SHALL stop the remaining stages for that Tick, record an error outcome identifying the failed stage, and continue to the next Tick without losing prior state.
6. WHEN a Tick completes, THE Angent SHALL update the loop state in ClickHouse with the outcomes of that Tick before the next Tick begins.
7. WHEN the improvement in target_metric falls below a configured progress threshold across a configured number of consecutive Ticks, THE Planner SHALL adapt the plan by changing at least one of thesis breadth, email angle, or send volume on the next Tick.
8. WHEN a Tick completes, THE Angent SHALL emit a state report containing the Tick outcomes and the progress toward the Goal.

### Requirement 3: Termination conditions

**User Story:** As an Investor, I want the loop to stop the moment my goal is met or my budgets are exhausted, so that Angent never overspends my time or email allowance.

#### Acceptance Criteria

1. WHEN a new Tick begins, THE Control_Loop SHALL evaluate the goal-met condition, the time-budget condition, and the email-budget condition before performing any other action in that Tick.
2. WHEN the target_metric of the Goal is achieved, THE Control_Loop SHALL stop within the current Tick and report the stop reason as goal-met.
3. WHEN the current time reaches or passes the deadline of the Goal, THE Control_Loop SHALL stop within the current Tick and report the stop reason as deadline-reached.
4. WHEN the number of emails sent reaches or exceeds the email_budget, THE Control_Loop SHALL stop within the current Tick and report the stop reason as email-budget-exhausted.
5. WHILE the number of emails sent is equal to the email_budget, THE Angent SHALL NOT send any further email.
6. IF two or more termination conditions are satisfied in the same Tick, THEN THE Control_Loop SHALL report a single stop reason selected in the priority order goal-met, then deadline-reached, then email-budget-exhausted.
7. WHEN the Control_Loop stops, THE Angent SHALL persist the final state and the stop reason in ClickHouse.
8. IF persisting the final state and the stop reason in ClickHouse fails, THEN THE Angent SHALL retry the persistence up to 3 times and, if all attempts fail, SHALL surface an error indicating that final-state persistence did not complete while retaining the final state and stop reason in memory.

### Requirement 4: Signal ingestion (GitHub and Hacker News, Hugging Face stretch)

**User Story:** As an Investor, I want Angent to discover newly-emerging startups from public signals, so that I see fresh deal flow I would not find on my own.

#### Acceptance Criteria

1. WHEN the Scanner runs a plan, THE Scanner SHALL retrieve candidate startups from GitHub through the Airbyte_Agent using repository, stargazer, and commit signals, limited to candidates whose first detected activity falls within the most recent 90 days, completing each Signal_Source request within 30 seconds.
2. WHEN the Scanner runs a plan, THE Scanner SHALL retrieve candidate startups from the Hacker News Algolia API using Show HN and Launch HN signals posted within the most recent 90 days, completing the request within 30 seconds.
3. WHEN the Scanner retrieves a candidate not already present in the `companies` table, THE Scanner SHALL write the candidate to the `companies` table in ClickHouse with a source attribution value identifying the originating Signal_Source.
4. IF a retrieved candidate matches an existing record in the `companies` table by its source-specific unique identifier, THEN THE Scanner SHALL update the existing record rather than create a duplicate, preserving the original creation timestamp.
5. IF a Signal_Source request fails or exceeds the 30-second timeout, THEN THE Scanner SHALL retry the request up to 3 times, and IF all attempts fail, THEN THE Scanner SHALL record a failure entry, continue with the remaining Signal_Sources, and complete the Tick.
6. WHERE the Hugging Face Hub source is enabled, THE Scanner SHALL retrieve AI organizations and models created on the Hugging Face Hub within the most recent 90 days, completing the request within 30 seconds.

### Requirement 5: Qualification against the thesis

**User Story:** As an Investor, I want each candidate scored and explained against my thesis, so that I only spend attention on genuine fits.

#### Acceptance Criteria

1. WHEN the Qualifier receives one or more new candidates, THE Qualifier SHALL produce, within 30 seconds per candidate, a numeric fit score in the range 0 to 100 (inclusive) for each candidate against the Thesis.
2. WHEN the Qualifier produces a fit score, THE Qualifier SHALL produce, using the TrueFoundry_Gateway, a natural-language explanation of the fit between 50 and 1000 characters (inclusive) that references the Thesis.
3. IF the TrueFoundry_Gateway fails to return an explanation within 30 seconds, THEN THE Qualifier SHALL store the fit score with a placeholder explanation indicating that the explanation is unavailable, and SHALL retain the fit score record.
4. WHEN the Qualifier completes a candidate, THE Qualifier SHALL store the fit score and the explanation on the candidate record in ClickHouse.
5. IF the write to ClickHouse fails, THEN THE Qualifier SHALL retry the write up to 3 times, and after the final failed attempt SHALL record an error indicating that persistence failed without discarding the computed fit score and explanation.
6. WHERE Pioneer is reachable and returns a fit score within 30 seconds, THE Qualifier SHALL obtain the fit score from Pioneer.
7. WHERE Pioneer is unreachable or fails to return a fit score within 30 seconds, THE Qualifier SHALL obtain the fit score from the Heuristic_Scorer.
8. WHEN a candidate's fit score meets or exceeds the configured qualification threshold (an integer from 0 to 100, inclusive), THE Qualifier SHALL pass that candidate to the Writer.
9. IF a candidate's fit score is below the configured qualification threshold, THEN THE Qualifier SHALL NOT pass that candidate to the Writer.

### Requirement 6: Self-improving scoring from reply outcomes

**User Story:** As an Investor, I want Angent's targeting to get better as real replies come in, so that quality rises over time instead of staying flat.

#### Acceptance Criteria

1. WHEN the Optimizer collects reply and open outcomes, THE Optimizer SHALL store each outcome against the corresponding email and company records in ClickHouse within 5 seconds of collection.
2. IF storing an outcome in ClickHouse fails, THEN THE Optimizer SHALL retry the store up to 3 times and, if all retries fail, SHALL retain the unstored outcome for the next collection cycle and produce an error indication identifying the affected outcome.
3. WHEN new outcomes are stored, THE Optimizer SHALL provide all newly stored outcomes to the active scorer as learning signal before the next Tick begins.
4. WHERE Pioneer is available, THE Optimizer SHALL update the Pioneer scoring model with the new outcomes.
5. IF the Pioneer scoring model update fails, THEN THE Optimizer SHALL retain the previous scoring model, continue scoring with the previous model, and produce an error indication identifying the failed update.
6. WHEN the scoring model is updated with outcomes indicating positive replies, THE Qualifier SHALL apply the updated scoring weights when ranking companies on subsequent Ticks.
7. WHEN the Planner reflects after outcomes are collected, THE Planner SHALL incorporate the observed reply-rate into the next Tick's plan.
8. WHEN an outcome collection cycle completes, THE Angent SHALL compute the reply-rate as the ratio of replies received to emails sent during the run and persist it to ClickHouse so that the metric trend is observable across all Ticks of the run.

### Requirement 7: Pluggable and optional Pioneer scorer with fallback

**User Story:** As the Builder, I want Pioneer to be optional behind a stable interface, so that the loop runs end-to-end even though the Pioneer API key arrives late.

#### Acceptance Criteria

1. THE Angent SHALL expose a single scoring interface, with identical method signatures and return structure, that both Pioneer and the Heuristic_Scorer implement.
2. WHEN Angent starts and Pioneer credentials are absent, THE Angent SHALL run using the Heuristic_Scorer and SHALL record an indication that it is operating in Heuristic_Scorer mode.
3. IF a Pioneer request does not return a successful response within 10 seconds during a Tick, THEN THE Qualifier SHALL fall back to the Heuristic_Scorer for that candidate, SHALL record an indication that fallback occurred for that candidate, and SHALL continue processing the remaining candidates in that Tick without aborting the Tick.
4. WHEN Pioneer credentials become present and a Pioneer request returns a successful response, THE Angent SHALL use Pioneer for subsequent candidates while keeping the Qualifier and Optimizer interface signatures and return structures unchanged.

### Requirement 8: Email drafting

**User Story:** As an Investor, I want personalized outreach drafts for qualified companies, so that each email reflects the specific company and my thesis.

#### Acceptance Criteria

1. WHEN the Writer receives one or more qualified candidates, THE Writer SHALL draft exactly one personalized email per candidate, up to the remaining email_budget, using the TrueFoundry_Gateway.
2. WHEN the Writer drafts an email, THE Writer SHALL incorporate the candidate's signals and the current plan's email angle into the email content.
3. WHEN the Writer completes a draft, THE Writer SHALL store the draft in the `emails` table in ClickHouse marked as unsent and unapproved.
4. WHILE the remaining email_budget for the current Tick is greater than zero, THE Writer SHALL produce a cumulative number of drafts in that Tick that does not exceed the remaining email_budget.
5. IF the remaining email_budget for the current Tick is zero, THEN THE Writer SHALL produce no drafts in that Tick.
6. IF the TrueFoundry_Gateway fails to return a draft for a candidate within 30 seconds or returns an error, THEN THE Writer SHALL skip that candidate without consuming email_budget, retain any successfully completed drafts, and record an indication that drafting failed for that candidate.
7. IF storing a completed draft in the `emails` table in ClickHouse fails, THEN THE Writer SHALL retry the store operation up to 3 times and, if all attempts fail, record an indication that the draft was not persisted.

### Requirement 9: Governance and human-approval gate

**User Story:** As an Investor, I want to approve every send and have hard caps enforced, so that nothing goes out without my consent and budgets are never exceeded.

#### Acceptance Criteria

1. WHILE a draft has not been marked approved by the Investor, THE Sender SHALL NOT send the draft and SHALL retain the draft in its unsent state.
2. WHEN the Investor approves a draft, THE Governance_Gate SHALL mark that specific draft as approved and SHALL permit only that approved draft to be sent.
3. IF a draft is modified after being marked approved, THEN THE Governance_Gate SHALL revert the draft to the unapproved state and SHALL require a new Investor approval before sending.
4. IF sending a draft would cause the cumulative number of sent emails to exceed the email_budget, THEN THE Governance_Gate SHALL block the send, SHALL record the block reason, and SHALL return an indication to the Investor that the send was blocked due to email_budget exhaustion.
5. WHILE the count of emails sent within the current rate-limit window is greater than or equal to the configured send rate limit, THE Governance_Gate SHALL defer the send until the start of the next rate-limit window.
6. WHEN a send is deferred for longer than 3600 seconds, THE Governance_Gate SHALL record the deferral reason and SHALL return an indication to the Investor that the send remains pending due to the rate limit.
7. THE Guild_Orchestrator SHALL route every send request through the Governance_Gate and SHALL reject any send that has not been permitted by the Governance_Gate.

### Requirement 10: Email sending with SMTP default and Gmail alternate

**User Story:** As the Builder, I want the loop to send via SMTP by default and optionally via an Airbyte Gmail connector, so that sending works regardless of the Airbyte tier.

#### Acceptance Criteria

1. THE Angent SHALL define a Sender_Interface that both SMTP_Sender and Gmail_Agent_Sender implement, exposing a send operation that accepts an approved draft and returns either a success result with a send timestamp or a failure result with a failure reason.
2. WHEN an approved draft is sent and no sender backend is explicitly selected, THE Sender SHALL use the SMTP_Sender as the default backend.
3. WHEN a sender backend reports a successful send, THE Sender SHALL update the corresponding `emails` record to the sent state and set the send timestamp to the time the success result was returned.
4. WHERE the Airbyte Gmail tier is unlocked and the Gmail_Agent_Sender is selected, THE Sender SHALL send approved drafts through the Gmail_Agent_Sender using the same Sender_Interface.
5. IF a sender backend does not return a success result within 30 seconds of the send operation starting, THEN THE Sender SHALL treat the send as failed, record the failure reason on the corresponding `emails` record, and leave the draft in an eligible-for-retry state.
6. IF the selected sender backend returns a failure result for a draft, THEN THE Sender SHALL record the failure reason on the corresponding `emails` record, leave the draft eligible for retry, and not decrement the email_budget for that attempt.
7. IF a draft has failed to send on 3 consecutive attempts, THEN THE Sender SHALL mark the `emails` record as failed and remove the draft from the eligible-for-retry state.

### Requirement 11: Loop orchestration and Guild fallback

**User Story:** As the Builder, I want the Python loop to run end-to-end on its own with Guild wrapping governance on top, so that a Guild problem never blocks the demo.

#### Acceptance Criteria

1. WHEN the Control_Loop is started, THE Python Control_Loop SHALL execute all six agents in sequence and complete without invoking the Guild_Orchestrator, where completion is defined as every agent returning a success status.
2. IF any one of the six agents fails during execution, THEN THE Python Control_Loop SHALL halt the sequence, retain the outputs of the agents that completed, and return an error result identifying the failed agent.
3. WHERE the Guild_Orchestrator is deployed, THE Guild_Orchestrator SHALL orchestrate the Control_Loop and call the Python backend over HTTP.
4. WHERE the Guild_Orchestrator is deployed, THE Guild_Orchestrator SHALL block any send that is not human-approved, reject any send that would exceed the configured send rate limit, and reject any send that would exceed the configured email-budget cap.
5. WHILE the Guild_Orchestrator is unavailable, THE Angent SHALL run the Control_Loop and enforce the Governance_Gate within the Python core.
6. IF the Guild_Orchestrator does not return a response within 5 seconds of a request, THEN THE Angent SHALL classify the Guild_Orchestrator as unavailable and continue the Control_Loop using the Python core.

### Requirement 12: Shared state and outcome analytics in ClickHouse

**User Story:** As the Builder, I want all agents to read and write a shared ClickHouse blackboard, so that state and analytics are consistent across the loop.

#### Acceptance Criteria

1. WHEN the Angent identifies a candidate company, THE Angent SHALL store the company record in the `companies` table in ClickHouse.
2. WHEN the Angent produces an email, THE Angent SHALL store the email record in the `emails` table in ClickHouse.
3. WHEN any agent updates loop state, THE agent SHALL persist the update to ClickHouse within 2 seconds, such that any subsequent read by another agent returns the updated state.
4. WHEN outcome analytics are requested, THE Angent SHALL compute reply rate from the ClickHouse `emails` table, aggregated per 24-hour interval across the full run duration.
5. WHEN the run starts in demo mode, THE Angent SHALL load seeded historical reply data into ClickHouse before the first Tick.
6. IF a write to or read from ClickHouse fails, THEN THE agent SHALL retry the operation up to 3 times and, if all retries fail, return an error indicating the ClickHouse access failure while retaining the unwritten data for a subsequent attempt.

### Requirement 13: Observability with Langfuse

**User Story:** As the Builder, I want every agent step and LLM call traced, so that I can show the loop's reasoning during the demo.

#### Acceptance Criteria

1. WHEN the Angent executes a step, THE Angent SHALL record a Langfuse trace for that step containing the step identifier, input payload, output payload, and start and end timestamps, within 2 seconds of step completion.
2. WHEN an LLM call is made through the TrueFoundry_Gateway, THE Angent SHALL record the call in Langfuse as a span linked to the current step trace, including the request prompt, the response content, and the token count.
3. IF Langfuse is not configured at Control_Loop startup, THEN THE Angent SHALL continue running the Control_Loop without tracing and SHALL produce a log entry indicating that tracing is disabled.
4. IF recording a Langfuse trace or span fails after 3 retry attempts, THEN THE Angent SHALL continue executing the current Control_Loop step without interruption and SHALL produce a log entry indicating the tracing failure.

### Requirement 14: OpenUI runtime generative UI

**User Story:** As an Investor, I want to refine my thesis in a conversational UI and open an agent-generated deep-dive on any company, so that the interface itself is generated by the agent for the moment.

#### Acceptance Criteria

1. WHEN the Investor sends a message to refine the Thesis, THE OpenUI_Surface SHALL render the thesis-refinement chat interface generated at runtime via OpenUI Lang within 5 seconds of the message being submitted.
2. IF runtime generation of the thesis-refinement chat interface fails or exceeds 5 seconds, THEN THE OpenUI_Surface SHALL retain the last successfully rendered interface state and present an error indication that generation did not complete.
3. WHEN the Investor opens a company for a deep dive, THE OpenUI_Surface SHALL render the deep-dive view generated at runtime via OpenUI Lang within 5 seconds of the open action.
4. IF runtime generation of the deep-dive view fails or exceeds 5 seconds, THEN THE OpenUI_Surface SHALL retain the prior view and present an error indication that the deep-dive view could not be generated.
5. THE OpenUI_Surface SHALL present agent-decided content in which every rendered element is produced by the OpenUI Lang generation step for the current request, and SHALL contain no element that is hardcoded independent of the agent output.
6. THE OpenUI_Surface SHALL render at least one element that accepts Investor input or triggers an Investor-initiated action in each generated view.

### Requirement 15: Plain React dashboard and Impeccable separation

**User Story:** As the Builder, I want the dashboard cards to be plain React that Impeccable can polish, while the OpenUI-generated components stay untouched, so that the OpenUI prize and the polish pass do not conflict.

#### Acceptance Criteria

1. THE React_Shell SHALL implement the qualified-company cards, the drafted-email preview, and the loop-status display as plain React components that are not generated by OpenUI and contain no OpenUI Lang markup.
2. THE React_Shell SHALL maintain a clear separation between plain React components and OpenUI-generated OpenUI Lang components such that each rendered component is unambiguously identifiable as exactly one of the two categories.
3. WHEN the Impeccable tool runs a polish pass, THE Impeccable tool SHALL apply styling and markup changes only to the plain React components that make up the React_Shell.
4. THE Impeccable tool SHALL NOT add, remove, restyle, or otherwise alter any OpenUI-generated OpenUI Lang component or its markup.
5. IF the Impeccable tool encounters an OpenUI-generated OpenUI Lang component during a polish pass, THEN THE Impeccable tool SHALL leave that component unchanged and continue processing the remaining plain React components without aborting the pass.

### Requirement 16: Front-end optional for the demo

**User Story:** As the Builder, I want to be able to demo from backend logs if the front end is unfinished, so that the front end is an enhancement and not a blocker.

#### Acceptance Criteria

1. WHEN the Control_Loop runs, THE Angent SHALL emit to backend logs and console output a record for each of the following stages: progress, qualifications, drafts, sends, and reply-rate trend, with each record containing a timestamp and a stage identifier.
2. WHEN any single Control_Loop stage completes, THE Angent SHALL write its corresponding log and console record within 2 seconds of stage completion.
3. WHERE the front end is disabled or unreachable, THE Angent SHALL execute every Control_Loop stage (progress, qualifications, drafts, sends, and reply-rate trend) end-to-end using only backend logs and console output, without requiring any front-end interaction.
4. IF the Angent cannot write a log or console record for a Control_Loop stage, THEN THE Angent SHALL continue executing the remaining Control_Loop stages and emit an error record to console output indicating which stage's record failed to write.

### Requirement 17: Single-builder ownership and critical-path scope

**User Story:** As the Builder, I want clear ownership and a critical-path-first build order encoded in the spec, so that the runnable core stays separated from the enhancements and I always have something that runs.

#### Acceptance Criteria

1. THE Angent specification SHALL assign every component — Python signal ingestion, ClickHouse schema and state, the six agents, the orchestration loop and termination logic, TrueFoundry integration, Pioneer scoring, the Sender, the Guild orchestrator, the OpenUI_Surface, the React_Shell, the Impeccable polish pass, Langfuse setup, the ClickHouse demo seed data, the Publisher, the x402 Payment_Gate, and the Devpost writeup — to the Builder as the single owner.
2. THE Angent specification SHALL define a strict critical-path-first build order so the Builder completes the runnable Python core (signal ingestion through Sender output to termination) BEFORE the enhancement layers (OpenUI_Surface, React_Shell, Impeccable, Langfuse, Publisher, Payment_Gate).
3. WHILE any enhancement layer is incomplete or non-functional, THE Angent SHALL still run the orchestration loop end to end on the Python core and reach termination with observable output.
4. THE Angent specification SHALL assign the Publisher, the Senso publication of the Deal_Memo to cited.md, and the x402 Payment_Gate that monetizes deal-memo fetches to the Builder.
5. THE Angent specification SHALL assign the SENSO_API_KEY, X402_FACILITATOR_URL, X402_NETWORK, X402_PAY_TO_ADDRESS, X402_PRICE, and EVM_PRIVATE_KEY environment configuration, together with the Publisher and Payment_Gate implementation tasks, to the Builder.

### Requirement 18: Sponsor real-work mapping

**User Story:** As the team, we want each sponsor technology to perform genuine work in the loop, so that the integrations are substantive rather than cosmetic.

#### Acceptance Criteria

1. WHEN the Angent performs GitHub discovery, THE Angent SHALL invoke the Airbyte_Agent to retrieve candidate records.
2. WHERE the Gmail sending tier is unlocked, WHEN the Angent sends an email, THE Angent SHALL route the send through the Airbyte_Agent.
3. WHEN the Angent performs LLM reasoning or generates email content, THE Angent SHALL route the request through the TrueFoundry_Gateway.
4. WHERE Pioneer is available, WHEN the Angent computes a qualification score, THE Angent SHALL obtain the self-improving qualification score from Pioneer.
5. IF Pioneer is unavailable when a qualification score is required, THEN THE Angent SHALL apply a documented fallback scoring path, record an indication that the fallback was used, and continue the loop without aborting.
6. WHEN the Angent reads or writes shared state or records outcome analytics, THE Angent SHALL use ClickHouse as the store of record.
7. WHEN the Angent advances the loop or evaluates a governance decision, THE Angent SHALL use the Guild_Orchestrator for orchestration and the Governance_Gate for the decision.
8. WHEN the Angent presents the runtime generative thesis chat or deep-dive interface, THE Angent SHALL render it through the OpenUI_Surface.
9. WHEN the Angent executes an agent step or issues an LLM call, THE Angent SHALL emit a corresponding trace to Langfuse that links the step or call to its loop run.
10. IF a sponsor technology invocation fails or times out, THEN THE Angent SHALL surface an error indication identifying the affected technology, preserve the prior loop state, and halt that step without corrupting shared state.
11. WHEN the Angent publishes the Deal_Memo, THE Angent SHALL publish it to cited.md through Senso.
12. WHEN an agent fetches the published Deal_Memo, THE Angent SHALL require and settle payment through the x402 Payment_Gate.
13. IF Senso is unreachable when the Angent publishes the Deal_Memo, THEN THE Angent SHALL write the Deal_Memo to a local file and surface the error without aborting the Control_Loop.
14. IF the Facilitator is unreachable when an agent fetches the published Deal_Memo, THEN THE Angent SHALL deny access and surface the error without serving the content unpaid.

### Requirement 19: Demo behavior

**User Story:** As the Builder, I want a small, fast, self-correcting demo run that judges can watch live, so that the self-improvement and governance are visible in minutes.

#### Acceptance Criteria

1. WHEN the demo Goal is started, THE Angent SHALL run with a target_metric between 0 and 1, a deadline of 120 seconds, an email_budget of 8 emails, and a Tick interval between 3 and 10 seconds.
2. WHEN the demo runs, THE Scanner SHALL pull at least 1 real candidate from GitHub and at least 1 real candidate from Hacker News before the deadline elapses.
3. IF the Scanner cannot retrieve at least 1 candidate from GitHub or Hacker News within 30 seconds, THEN THE Scanner SHALL continue the run with the candidates already retrieved and record an indication that the source was unavailable.
4. WHEN a draft is approved during the demo, THE Sender SHALL deliver the email to the controlled inbox within one Tick of approval.
5. IF a send during the demo would cause the cumulative sent count to exceed the email_budget of 8 emails, THEN THE Governance_Gate SHALL block the send, leave the cumulative sent count unchanged, and display an indication that the send was blocked for exceeding the email_budget.
6. WHILE the demo run proceeds, THE Angent SHALL display the current reply-rate metric updated at least once per Tick, where the reply-rate value increases relative to the prior Tick as the scorer learns from seeded outcomes.
7. WHEN the demo Goal target_metric is met, THE Control_Loop SHALL self-terminate within one Tick and display the stop reason indicating the Goal was met.
8. WHEN the deadline of 120 seconds elapses or the email_budget of 8 emails is exhausted, THE Control_Loop SHALL self-terminate within one Tick and display the stop reason indicating the exhausted deadline or budget.
9. WHEN the demo presents learning results, THE Angent SHALL display a statement that learning runs on seeded historical outcomes.
10. WHEN the demo completes, THE Angent SHALL make the Langfuse trace of the run available for display within 10 seconds of termination.
11. WHEN the demo run terminates, THE Angent SHALL publish a live Deal_Memo to cited.md via Senso and display the resulting cited.md URL.
12. WHEN the demo demonstrates monetization, THE Angent SHALL show a buyer agent receiving an HTTP 402 Payment Required response, paying approximately 0.001 testnet USDC through the x402 Payment_Gate, and then receiving the Deal_Memo content.

### Requirement 20: Publishing the deal memo to cited.md

**User Story:** As an Investor, I want Angent to publish its qualified-company findings to cited.md, so that my agent's output is citable on the open web with provenance.

#### Acceptance Criteria

1. WHEN the Optimizer completes a Tick or the run reaches its end, THE Publisher SHALL serialize the run's qualified companies, including name, URL, source, fit_score, fit_explanation, and signals, from ClickHouse into a Deal_Memo markdown document.
2. THE Deal_Memo SHALL include, for every company, a provenance citation linking to that company's real source URL, being the GitHub repository or the Hacker News post.
3. WHEN the Deal_Memo is ready, THE Publisher SHALL publish it to cited.md via the Senso CLI `senso engine publish` using SENSO_API_KEY.
4. WHEN publication succeeds, THE Publisher SHALL persist the returned cited.md URL and slug to ClickHouse.
5. IF Senso is unreachable or publication fails, THEN THE Publisher SHALL write the Deal_Memo markdown to a local file, record that the remote publish failed, and SHALL NOT block or abort the Control_Loop.
6. THE Deal_Memo published to cited.md SHALL be markdown and SHALL NOT be OpenUI Lang, the OpenUI deep-dive view defined in Requirement 14 SHALL render the same ClickHouse data for the in-app human surface, and the two output types SHALL remain separate.

### Requirement 21: Monetizing fetches with the x402 payment gate

**User Story:** As the Builder, I want fetches of the published deal memo to be charged through agent payment rails, so that Angent's output is monetized per fetch.

#### Acceptance Criteria

1. THE Payment_Gate SHALL protect the deal-memo fetch endpoint such that a request without a valid x402 payment receives an HTTP 402 Payment Required response.
2. WHEN a request includes a valid x402 payment, THE Payment_Gate SHALL settle the payment via the Facilitator and return the deal-memo content.
3. THE Payment_Gate SHALL be configured from the environment variables X402_FACILITATOR_URL, X402_NETWORK, X402_PAY_TO_ADDRESS, and X402_PRICE, using the "exact" scheme in USDC.
4. IF the Facilitator is unreachable, THEN THE Payment_Gate SHALL deny access with an error indication and SHALL NOT serve the content unpaid.
5. WHERE a buyer agent is demonstrated, THE buyer SHALL pay automatically using EVM_PRIVATE_KEY and then receive the content, consistent with the reference implementation at x402-test/buyer.mjs.


### Requirement 22: Continuous version control to GitHub

**User Story:** As the Builder, I want every change committed to main and pushed to GitHub immediately, so that I always have fine-grained version history and a recoverable, demo-ready repo.

#### Acceptance Criteria

1. THE Builder SHALL use the GitHub repository at https://github.com/Jiyungi/angent as the single remote, with `main` as the working branch.
2. WHEN the Builder completes any discrete change, being a new file, a function, a bug fix, a config edit, or a removed file, THE Builder SHALL create a Git commit on `main` that captures only that change with a descriptive message.
3. WHEN a commit is created on `main`, THE Builder SHALL push the commit to the GitHub repository before starting the next change.
4. THE Builder SHALL commit frequently and in small increments such that each implementation sub-task results in at least one commit pushed to `main`.
5. IF a push to GitHub fails, THEN THE Builder SHALL resolve the failure and retry the push before continuing further implementation work.
6. THE Repository SHALL exclude secrets from version control by keeping `.env` untracked via `.gitignore` while committing `.env.template`.

### Requirement 23: Lean codebase — no separate test or checking harness

**User Story:** As the Builder, I want to drop the heavyweight property-based test suite and ad-hoc checking scripts, so that one person can ship the end-to-end loop quickly and verify it by running the demo.

#### Acceptance Criteria

1. THE Angent codebase SHALL NOT include a property-based test suite, unit-test suite, integration-test suite, or smoke-test suite as a build deliverable.
2. THE Builder SHALL remove existing standalone checking and probe scripts that are not part of the runtime loop, specifically `test_airbyte.py` and `check_connectors.py`, from the Repository.
3. WHEN the Builder needs to verify behavior, THE Builder SHALL verify by running the demo Control_Loop end to end and observing the backend log and console output and the live cited.md and x402 demo, rather than by running an automated test suite.
4. THE Builder SHALL retain runtime code required by the loop, for example `angent/email_sender.py`, and the x402 reference implementation at `x402-test/` that the Payment_Gate reuses.
5. THE Angent specification SHALL NOT define correctness-property test tasks or "ensure all tests pass" checkpoints; verification checkpoints SHALL instead confirm that the loop runs and that changes have been committed and pushed to `main`.
